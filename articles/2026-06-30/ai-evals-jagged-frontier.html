The Jagged Frontier: Why Your AI Feature Needs an Eval, Not Just a Vibe Check

30 June 2026 · synthesised from 1 source + first-principles

AI capability is uneven across domains and tasks — what works brilliantly in one context fails unpredictably in another. Here's how to build a lightweight eval set so you know your AI feature actually works, instead of hoping it does.

In a recent conversation with Dwarkesh Patel, mathematician Grant Sanderson (3Blue1Brown) made an observation worth sitting with: AI has made far faster progress in mathematics than in most other fields — and even within maths, progress is uneven. Some sub-areas are essentially solved by current models; others remain hard. That unevenness is the rule, not the exception, and it's the single most useful idea to internalise if you're shipping AI features without a formal ML background.

The jagged frontier

Call it the jagged frontier: AI competence doesn't rise smoothly across domains, or even within one task category. A model can be near-superhuman at competition maths and unreliable at a task that looks simpler to a human — extracting a date from a messy invoice, say — because the skill distributions in training data, and how a task rewards exact correctness versus "plausible-sounding," don't line up with how hard the task looks from the outside. The boundary between "this just works" and "this fails in ways you won't predict" is invisible until you actually probe it.

Why this matters for you: when you vibecode an AI feature — a chatbot, a categoriser, a summary generator — you're implicitly betting your specific use case sits on the "works" side of that frontier. General benchmarks, the ones that justify a model's marketing, tell you almost nothing about where your particular prompt, in your particular domain, with your particular edge cases, actually lands.

The fix: evals, not vibes

The standard engineering answer is an eval — short for evaluation — a fixed, repeatable test set for an AI feature, the same way unit tests are a repeatable test set for ordinary code. "I tried it a few times and it looked good" isn't good enough: a handful of manual checks samples only the inputs you thought to try, doesn't survive someone editing the prompt next week, and tells you nothing when you swap to a cheaper model six months from now to cut costs.

A minimal eval set for an indie app looks like this:

  • 20–50 representative inputs — pulled from real usage if you have it, written by hand if you don't. Skew toward the edge cases that worry you: empty input, adversarial input, the weird formatting a real user will inevitably paste in.
  • A scoring method. For tasks with a clear right answer, exact match works. For open-ended output — a generated summary, a tone-matched reply — use an LLM-as-judge: a second model call that grades the first model's output against a rubric you write, calibrated against a small sample you grade by hand first so you trust the judge.
  • A re-run trigger. Run the eval set every time you touch the prompt, change the model, or change the surrounding code that builds the prompt. Track the score over time the way you'd track a build passing or failing.

Concretely: say you've built a feature that parses a free-text workout note ("3x8 bench 60kg") into structured sets. Your eval set is 30 real or synthetic notes — including ones with typos, mixed units (kg vs lb), supersets, and a blank field. You score "did it produce the correct structured output" as exact match, and run it before every prompt change. The first time you do this, you'll usually find your feature was silently failing on 10–20% of inputs you'd never tried by hand — that gap is the jagged frontier showing up in your own product.

What this buys you: the difference between "I shipped a feature that mostly works" and "I shipped a feature I can prove works, and I'll know within minutes if a change breaks it." That second sentence is what "robustness" means for an AI feature — it's not a different skill from the software fundamentals you're already building toward, it's the same discipline (repeatable tests, regression detection) applied to a non-deterministic component. Next time you ship an AI feature, write the eval set before you ship, not after something breaks in production.
