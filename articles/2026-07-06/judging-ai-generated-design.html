AI Can Draw The Screen Now — Your Job Is Judging Which One's Right

6 July 2026 · synthesised from 2 sources

Two design essays published this week converge on the same shift: once AI generates the screen and the code behind it, the designer's real work becomes judgment and explanation, not production.

Two design essays published this week land on the same point from opposite directions: AI can now generate the screen and the code behind it, but nobody has automated the judgment call of which output to trust, or how to explain it to the person asking why.

What AI took off your plate

UX Collective's Christine Vallaure breaks the Figma-to-code AI stack into four layers, and the breakdown matters because skipping one has a specific, predictable cost. The first layer is MCP (Model Context Protocol), the wiring that lets an assistant like Claude read a design file's real properties, spacing, colour tokens, component structure, instead of guessing from a screenshot. The second is markdown files that hold the "why" behind design decisions, so you stop re-explaining the same rule every session. The third is skills: repeatable workflows that run the same way each time rather than one-off guidance you retype. The fourth, and the one Vallaure calls critical, is Code Connect, the mapping layer that ties a Figma component to the code shipping in production.

Skip that mapping layer and you get what's worth naming on its own: design-code drift. You change a button's padding in Figma, the change never propagates because nothing but that mapping layer enforces the link, and three sprints later the design file and the running app have quietly stopped agreeing with each other. Vallaure's other point is just as useful for a solo builder: don't over-build the stack. A one-person project needs layers one and two; only a large team running a shared design system needs all four.

The explaining problem didn't disappear, it moved

Nielsen Norman Group's Katie Schmidt, Jessa Anderson and Hayley Mortin published a framework on 3 July mapping three roles inside a business that each need a different kind of explanation from the same AI system, using an AI help-desk agent as the example. A governance lead wants a global, static explanation: audit trails, edge-case handling, compliance documentation covering the whole system. A builder, the person configuring or debugging it, wants a local, interactive explanation tied to one specific output: "escalated because: account flagged for multi-factor review." A domain expert wants something in between: a local, static explanation anchored to precedent, like "based on 14 similar past tickets," plus a lightweight way to flag when it's wrong.

Hand a governance lead a builder's debug log and they'll disengage, because none of it maps to what they're accountable for. Hand a builder an audit summary and they can't fix anything, because there's no specific output to trace. The three explanations aren't three drafts of the same document — they're genuinely different information, tiered to what each reader needs to act.

It's the hierarchy skill you already have

This is the same instinct behind visual hierarchy, just applied one layer up. Deciding what a screen's primary, secondary and tertiary information is, what the eye lands on first, what waits, is the same decision as deciding what a governance lead needs first versus what a builder needs first. The skill isn't new; the object you're applying it to is. Sorting an explanation by audience is an information-hierarchy problem before it's a writing problem, and you've already built the muscle for solving those.

Try this on your next AI-assisted build

Next time an AI assistant generates a screen or feature for one of your own apps, name which of Vallaure's four layers you're touching, wiring, rules, workflow or mapping, before you decide how much scaffolding it deserves. And before you ship anything that makes a decision a user might question, write one line for each audience who could ask why: what would the audit view, the debug view and the plain-language view of this decision look like. That forces the judgment call up front — instead of shipping whatever AI drafted first and explaining it later under pressure.
