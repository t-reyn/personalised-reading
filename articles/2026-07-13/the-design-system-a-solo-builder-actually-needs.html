The Design System a Solo Builder Actually Needs

13 July 2026 · synthesised from 2 sources

Design systems sound like a big-company problem, but the two dimensions that stop an app looking inconsistent — tokens and governance — fit in one file, even for a team of one.

Every app that grows past a handful of screens starts to drift: one screen's button is 8px more rounded than another's, a heading is 22px on one page and 24px on the next, a "warning" colour shows up as three different shades of amber. Nielsen Norman Group's design-system maturity framework, published this month, breaks the fix into six dimensions — organisational alignment, team effectiveness, infrastructure, governance, support and adoption — scored independently on a 1-to-5 scale rather than as one linear ladder. Most of those dimensions assume a company with multiple design and engineering teams. Two of them don't, and they're the two that actually stop the drift: infrastructure and governance.

Infrastructure: design tokens

Design tokens are the mechanism, and they're simpler than the name suggests. A token is a named, reusable value — a colour, a spacing amount, a type size — that every screen references instead of a hard-coded number. Rather than one screen using #3B82F6 and another using #3A82F5 for the same "primary" blue, both reference a single token called something like color-primary. Change the token once, and every screen that uses it updates together. This is the same vocabulary you've already got — visual hierarchy, a type scale, whitespace rhythm, contrast for readability — just packaged as variables instead of left to memory. You already know a type scale should step predictably (say 14 / 16 / 20 / 28px) and that hierarchy comes from consistently applying size, weight and spacing together, not randomly per screen; a token file is what turns "I try to keep it consistent" into "it's structurally impossible to be inconsistent," because there's only one number to pull from. The same discipline applies to contrast: pick your text-on-background pairs once, token them, and every new screen inherits a pairing you've already checked is readable, rather than re-guessing it each time.

Governance: the rules AI (or a future you) actually follows

NN/g's governance dimension is about who can add or change something in the system and how exceptions get handled — sounds corporate, but it maps directly onto a problem solo builders hit constantly. Patrick Neeman, writing in UX Collective this month, makes the case that AI tools have shifted design craft away from producing pixels and toward judgement — deciding which of a thousand plausible AI-generated options actually deserves to exist. His practical fix is a standing "scaffolding" document (he cites Google's internal DESIGN.md format) that states the rules explicitly, because — his phrase — "the machine cannot read your hands; it can only read your words." For a one-person team, governance doesn't need a review board. It needs one short file: your token values, your spacing scale, three or four sentences on when a new colour or component is allowed versus when you should reuse what exists. That file is what an AI coding assistant reads before generating a new screen, and it's what stops "just this once" exceptions from quietly becoming the new default three features later.

Skip the other four dimensions — for now

NN/g's organisational-alignment, team-effectiveness, support and adoption dimensions matter once a system has to survive multiple contributors and leadership turnover. They're close to irrelevant to a single builder shipping fast — there's no one else's adoption to drive, no leadership sponsorship to secure. Trying to build all six dimensions before shipping a feature is how design systems become a procrastination device. A maturity-2 system — a token file and a one-page rules doc, both genuinely used — beats a maturity-5 framework that exists only as an aspiration.

What to do with the next screen you build

Open a single file and write down your actual tokens: two or three colours (primary, background, one semantic warning/error/success set), three or four type sizes, and a spacing scale of four or five steps. Add three sentences on when you're allowed to introduce a new value versus reuse an existing one. Reference that file — or paste it into context — every time you or an AI assistant builds a new screen. That's the whole system. It won't win a design award, but it's the difference between an app that looks like one product and one that looks like four screens stitched together.
